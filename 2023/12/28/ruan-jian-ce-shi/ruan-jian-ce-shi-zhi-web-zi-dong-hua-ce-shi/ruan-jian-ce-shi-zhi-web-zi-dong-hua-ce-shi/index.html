软件测试之web自动化测试

一. web自动化入门

1.什么是自动化测试

概念：让程序代替人工去验证系统功能的过程

2.自动化测试能解决什么问题

2.1 解决回归测试

回归测试：项目在发布新版本之后对项目之前的功能进行验证

2.2 解决压力测试

压力测试：可以理解多用户同时去操作软件，统计软件服务器处理多用户的请求能力

2.3 解决兼容性测试

兼容性测试：不同浏览器（浏览器，分辨率，操作系统）

2.4 提高测试效率，保证产品质量

3.自动化测试优点

3.1 在最短时间内运行更多的测试用例
3.2 自动化脚本可重复运行
3.3 减少人为错误
3.4 克服手工测试的局限性

4.自动化测试分类

4.1 web自动化测试
4.2 移动自动化测试
4.3 接口自动化测试
4.4 单元自动化测试

5.什么是web自动化测试

概念：让程序代替人工自动验证web项目功能的过程

6.什么web项目适合做自动化测试

6.1 需求变动不频繁
6.2 项目周期长
6.3 项目需要回归测试

7.web自动化测试所属分类（代码可见度角度）

7.1 黑盒测试（功能测试）
7.2 白盒测试（单元测试）
7.3 灰盒测试（接口测试）
7.4 web自动化测试属于黑盒测试

8.主流的web自动化测试工具

8.1 QTP是一个商业化的功能测试工具，收费，支持web，桌面张东海测试
8.2 Selenium是一个开源的web自动化测试工具，免费，主要做功能测试
8.3 Robot framework是一个基于Python可扩展的关键字驱动的测试自动化框架

二 . Selenium-API操作

1.什么是Selenium

Selenium是一个用于web应用程序的自动化测试工具，中文意思为硒

2. Selenium特点

2.1 开源软件，免费
2.2 跨平台（Linux，windows，mac）
2.3 支持多种浏览器（谷歌，火狐等）
2.4 支持多语言（java，python等）
2.5 成熟稳定，已被谷歌等公司广泛使用
2.6 功能强大，能够实现类似商业工具的大部分功能，可实现定制化

3.元素定位相关工具

3.1 为什么要用元素定位

要使用web自动化操作元素，必须找到此元素

3.2 定位工具

火狐，谷歌

4.Selenium元素定位方式

4.1 id（基于元素id属性定位方式）

说明：id在页面中唯一

element=driver.find_element_by_id(id)
4.2 name（基于元素anme属性定位方式）

说明：元素要有name属性

element=driver.find_element_by_name(name)
4.3 class_name（基于元素class属性定位方式）

说明：元素要有class属性

element=driver.find_element_by_class_name(class_name)
4.4 tag_name（基于元素标签定位方式）

说明：元素一般有多个，默认返回第一个标签元素

element=driver.find_element_by_tag_name(tag_name)
4.5 link_text（基于超链接文本定位方式）

说明：定位超链接标签，只能精准匹配

element=driver.find_element_by_link_text(link_text)
4.6 partial_link_text（基于超链接文本定位方式）

说明：定位超链接标签，可以使用精准匹配或模糊匹配，如果使用模糊匹配最好使用能代表唯一的关键字

如有多个值，默认返回第一个

element=driver.find_element_by_partial_link_text(partial_link_text)
4.7 XPath（基于元素路径定位方式）
4.7.1路径

绝对路径：以单斜杠开头逐级编写，不能跳级，如：/html/body/div/p[1]/input

相对路径：以双斜杠开头，双斜杠后面跟标签名称，不知标签名称可使用星号代替，如//input，//*

4.7.2 路径结合属性

在xpath中，所有属性必须使用@符号修饰，如：//*[@id=’id值’]

4.7.3 路径结合逻辑（多个属性）

如：//*[@id=’id值‘and@name=’name’]

4.7.4 路径结合层级

如：//*[@id=’父级id值’]/input

4.7.8 XPath扩展

//*[text()=”xxxxx”]，定位文本值等于xxxxx的元素

//*[contains(@attribute，”xxxxx”)]，属性中含有xxxxx的元素

//*[starts-with(@attribute，”xxxxx”)]，属性以xxxxx开头的元素

4.8 CSS（基于选择器定位方式）
4.8.1 CSS定位方法
element=driver.find_element_by_css_selector(css_selector)
4.8.2 CSS定位常用策略

id选择器

说明：工具元素id属性来选择

格式：#id

举例：#userA<选择id属性值为userA的元素>

class选择器

说明：工具元素class属性来选择

格式：.class

举例：.telA<选择class属性值为telA的所有元素>

元素选择器

说明：工具元素的标签名称来选择

格式：element

举例：input<选择所有input元素>

属性选择器

说明：工具元素的属性来选择

格式：[属性名=’属性值’]

举例：[username=’username’]

层级选择器

说明：根据标签层级来选择

格式：p>input，p input，提示：>与空格的区别，大于号必须为子元素，空格则不用

举例：p>input[placeholder=’电子邮箱’]

4.8.3 CSS扩展

input[type^=’p’]，type属性以p字母开头的元素

input[type$=’d’]，type属性以d字母结束的元素

input[type*=’w’]，type属性包含w字母的元素

4.8.4 Xpath与CSS类似功能对比
定位方式 Xpath CSS
元素名 //input input
id //input[@id=’userA’] #userA
class //*[@class=’telA’] .telA
属性 //[text()=”xxxxx”]，//[contains(@attribute，”xxxxx”)]，//*[starts-with(@attribute，”xxxxx”)] input[type^=’p’]，input[type$=’d’]，input[type*=’w’]

5.1 元素等待
5.1.1 什么是元素等待

概念：在定位页面元素是如果未找到，会在指定时间内一直等待的过程

5.1.2 为什么要元素等待

由于电脑配置或网络原因，在查找元素时，元素代码未在第一时间内被加载出来，抛出未找到元素异常

5.1.3 元素等待类型

显示等待：定位元素时，如能定位到元素则直接返回该元素，不触发等待；如果不能定位到该元素，则间隔一段时间后再去定位元素；如果在达到最大时长还没有找到该元素，则抛出元素不存在异常

webDriverwait(driver,timeout,poll_frequency=0.5)

调用方法

until(method)

一般用匿名函数实现

webDriverwait(driver,10,1).until(lambda x:find_element_by_id('userA'))

说明：显示等待针对单个元素

隐示等待：定位元素时，如能定位到元素则直接返回该元素，不触发等待；如果不能定位到该元素，则间隔一段时间后再去定位元素；如果在达到最大时长还没有找到该元素，则抛出元素不存在异常

driver.implicitly_wait(timeout)，timeout为等待时长，单位为秒

说明：隐式等待为全局设置，针对所有元素

5.8 cookie介绍
5.8.1 cookie概念

cookie由服务器生成，并且保存在用户浏览器上的小文本文件，可以包含用户相关的信息；cookie数据格式为键值对组成；客户端请求服务器，如果服务器需要记录该用户状态，就向客户端浏览器颁发一个cookie数据；当浏览器再次请求该网站时，浏览器把请求的数据和cookie数据一同提交给服务器，服务器检查该cookie以此来辨认用户状态；

5.8.2 selenium操作cookie
获取指定cookie，get_cookie(name)，name为cookie的名称
获取本网站所有本地cookies，get_cookies()
添加cookie，add_cookie(cookie_dict)，cookie_dict：一个字典对象，"name"and"value"
5.9 截屏操作
5.9.1 窗口截图方法
driver.get_screenshot_as_file(imgpath)，imgpath：图片保存路径

多条用例执行失败，会产生多张图片，可以采用时间戳的形式区分

driver.get_screenshot_as_file("../image/%s.jpg"%(time.strftime("%Y_%m_%d_%H_%M_%S")))

三. UnitTest框架

1.1 什么是UnitTest框架

概念：UnitTest是python自带的一个单元测试框架，可用它来做单元测试

1.2为什么使用UnitTest框架

能够组织多个用例去执行
提供丰富的断言方法
能够生成测试报告

1.3 UnitTest核心要素

1.3.1 TestCase，测试用例
1.3.2 TestSuite，测试套件
1.3.3 TextTestRunner，以文本的形式运行测试用例
1.3.4 TestLoader，批量执行测试用例，搜索指定文件夹内指定字母开头的模块
1.3.5 Fixture，两个固定函数，一个初始化使用，一个结束时使用

1.3 TestCase测试用例使用步骤

1.3.1 导包 import unittest
1.3.2 新建测试用例并继承 unittest.TestCase
1.3.3 测试方法必须以test字母开头

1.4 TestSuite使用步骤

1.4.1 概念：测试套件，多条测试用例集合在一起，就是一个TestSuite
实例化：suite=unittest.TestSuite()
添加用例：suite.addTest(CLassName("MethodName"))
添加扩展：suite.addTest(unittest.makeSuite(ClassName))
提示：TestSuite需要配合TestRunner才能被执行
